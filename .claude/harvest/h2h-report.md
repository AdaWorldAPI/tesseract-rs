# Head-to-Head: tesseract-rs vs the C++ `tesseract` CLI

Two harnesses, one corpus. The point is a **defensible** comparison — matched
inputs, matched threading, matched load amortization, and the recognizer
isolated from the line finder. Where the comparison crosses an untranscoded
stage, it is named, never blurred.

- **Corpus:** the HierText validation **legible-horizontal-Latin line** subset
  (`corpus/hiertext/`, generated by `gen_hiertext_lines.py` — 2286 lines from 60
  images; drop tallies printed by the generator: illegible 1401, rotated 1075,
  tiny 238, vertical 27, non-Latin 22). Scene text at large rotation / illegible
  is dropped: it measures the line finder, not recognition, and defeats BOTH
  engines.
- **Model:** the committed `corpus/model/eng.lstm*` (Apache-2.0 tessdata),
  identical bytes both sides.
- **Machine:** x86_64, AVX-512 VNNI (no AMX → our int8 GEMM runs ndarray's
  Tier-2 VPDPBUSD path). Idle. Release build both sides.

---

## 1. Quality — recognition only (both engines fed the IDENTICAL GT-boxed crop)

`h2h_compare` crops each GT line box, hands the SAME grey crop to our
`recognize_grey_line` (dict on) AND to `tesseract --psm 7 --oem 1 -l eng`
(pinned, `OMP_THREAD_LIMIT=1`), scores both against the GT transcription. This
**isolates the recognizer from the line finder** — the page-level segmentation
deficit (E-OCR-LINEFINDER-1 boundary; Phase-7 ColumnFinder) is out of the loop.

**Aggregate (2286 lines, 0 skipped):**

| metric | tesseract-rs | C++ CLI |
|---|---|---|
| mean CER vs GT (case-sensitive) | 0.4612 | **0.3664** |
| mean CER vs GT (case-folded) | 0.4352 | — |
| mean WER vs GT | 0.6863 | **0.5738** |
| mean CER(ours vs cli) | 0.6906 | — |

**Attribution (exact case-sensitive equality vs GT):**

| bucket | count | % | meaning |
|---|---|---|---|
| both-correct | 695 | 30.4% | the shared model tier |
| both-wrong-same-way | 580 | 25.4% | a model/scene limit **neither** clears — NOT our defect |
| both-wrong-diff | 718 | 31.4% | both wrong, differently (scene noise, input conditioning) |
| **ours-only-wrong** | **232** | **10.1%** | our genuine gap on lines the CLI gets right |
| cli-only-wrong | 61 | 2.7% | **we beat the CLI** |

Top substitutions ours-vs-GT: `E→e`, `o→e`, `O→o` (case), `0→)`, `I→e` (shape on
hard glyphs). Edit ops: 4149 sub / 3039 del / 940 ins.

**Reading it honestly.** This corpus is SCENE text — HierText exists to motivate
detection-model pipelines, and it defeats classic OCR for BOTH engines (CER
~37–46%). The value is the **attribution**, and it reconciles cleanly with the
byte-parity claim:

- **56.8% of lines are both-wrong** (25.4% identically + 31.4% differently). The
  25.4% *identical* wrong is the eng.lstm model itself failing — same model both
  sides, so it is a MODEL limit, not a transcode gap.
- Where the CLI succeeds, we match it on **695/(695+232) = 75%** of lines; we are
  genuinely worse on **10.1%** and better on **2.7%**.
- **The 10.1% ours-only gap is largely INPUT CONDITIONING, not the recognizer
  core.** The recognizer math is byte-identical (P6: 8/8 model-height lines; here
  the crops feed the same proven pixScale→FromPix→forward→beam chain). What the
  CLI does that we DON'T transcode is `RecognizeLine`'s **auto-invert retry**
  (`lstmrecognizer.cpp:349-380`: if the line reads light-on-dark, it re-runs
  inverted and keeps the better result) plus its DPI/contrast line-prep. On
  scene text (frequent light-on-dark signage) that conditioning is exactly where
  the CLI pulls ahead — a documented un-transcoded PREPROCESSING branch, not a
  recognizer defect. Transcoding auto-invert is a bounded, byte-parity-able next
  leaf that would close most of the 10.1%.

So: on documents (P6) quasi-parity; on scene text both poor, and our gap to the
CLI is a missing preprocessing branch + the model's own scene ceiling — the
recognizer core stays byte-parity.

## 2. Speed — matched (single-thread both, load amortized once, same 300 crops)

The P6 perf report's per-page CLI number (5.38 pages/s) paid process startup +
traineddata reload on EVERY page — ~92% overhead our in-process harness never
pays. That number is NOT a valid H2H ratio. Here both sides amortize the model
load once and run single-threaded:

| metric | tesseract-rs | C++ CLI (batch) | ratio |
|---|---|---|---|
| lines/sec | **12.88** | 93.97 | CLI **7.3× faster** |
| peak RSS | **8.4 MB** | 38.3 MB | ours **4.6× leaner** |
| cold model-load | 0.5 ms | 98 ms | ours 196× faster to load |

Our side: `h2h_speed` (in-process, model loaded once, 3 timed passes, min, dict
path). CLI side: `run_cli_speed.py` (image-list batch mode = one process over
all crops, `OMP_THREAD_LIMIT=1`, peak child RSS via `getrusage`). Both on the
same 300 HierText line crops, both pinned to the committed model
(`--tessdata-dir` on the recombined `corpus/model` components, codex P2 — the
CLI runs the SAME bytes the Rust side loads, not the host's installed
`eng.traineddata`).

### Where our time goes (per-stage, our side)

| stage | µs/line | % of end-to-end |
|---|---|---|
| parse_pgm | 2.5 | 0.0 |
| prescale_grey_to_height | 68.8 | 0.1 |
| from_grey_pix (grid build) | 46.3 | 0.1 |
| **network.forward** | **~77000** | **99.4** |
| beam + extract + text (dict, derived) | ~800 | 0.6 |

(The beam row is the **dict** path — matched to the quality harness + the CLI's
default English data, codex P2. Dict decoding does more work than the plain
beam, but `forward` still dominates.)

**`network.forward` is 99.4% of the time.** The whole gap is the dense int8
LSTM GEMM over timesteps, single-threaded. This is NOT a compute-primitive
problem — the forward already runs ndarray's AVX-512 VNNI path (byte-parity
proven). It is an *engineering* gap: single-thread + per-line GEMM (under-filled
VNNI tiles on variable-width lines) + per-line allocations. The CLI is 20 years
profile-tuned and threads via OpenMP.

## 3. The honest verdict

- **As an OCR *engine* (line image → text): it IS the original.** Byte-identical
  where the chain is transcoded (P6: 8/8 + this recognition-only corpus).
- **Quality gap to the CLI is layout, not recognition** — Phase-7 ColumnFinder /
  tospace, the documented scope boundary. On matched line crops the recognizer
  is at parity (see §1).
- **Speed: 7.3× slower per line, 4.6× leaner in RAM, 196× faster to load.** The
  slowness is single-thread + un-batched forward, addressable without touching
  byte-parity (the math is identical): (1) Rayon over lines — lines are
  independent, `&self` compute, per-line seeded RNG → near-linear with cores,
  exactly how the CLI threads; (2) width-bucketed batch-forward to fill the
  VNNI/AMX tiles; (3) stage pipelining last. None changes a single output byte.

The leaner RAM + near-instant load make it the better fit for edge/SoC
(recognizer + ndarray NEON, contract-only, no lance brain-crates); the throughput
gap is where the server tier or the parallelization wave earns its keep.
