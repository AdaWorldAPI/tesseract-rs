## §0 — Decision: the Tesseract corpus pin

**Pinned: `tesseract-ocr/tesseract` tag `5.5.0`.**

Rationale:

- It is a **tagged release**, not `HEAD` — reproducible, the property the
  codegen plan's "stable IR dump" (D-OCR-40) depends on. A moving `HEAD` would
  make `CPP-AST-RT` (round-trip determinism) un-pinnable.
- It **aligns with the crate as it stands today** — the published `tesseract-rs`
  (the FFI wrapper still in `src/`) targets `tesseract-sys` 0.6 / libtesseract
  5.x, so the harvested corpus and the FFI oracle (`tesseract-rs-ast-dll-codegen-v1`
  D-OCR-42 diff-gate) speak the same Tesseract major.

**Advance policy:**

- Any **patch** bump within `5.5.x` is free — re-run the harvester, no IR-shape
  change expected (the predicate vocab is corpus-independent).
- A **minor/major** bump (`5.6`, `6.x`) MUST re-run `CPP-SCHEMA-FIT`
  (`ruff/.claude/plans/cpp-spo-probes-v1.md`) before landing, because new
  class-body constructs can fall outside the closed predicate vocab.

The pin is recorded HERE and echoed in every generated file's provenance header
(§1). If the operator prefers `HEAD` or a different tag, change this line and
the header template moves with it.

## §1 — Provenance header (every generated `.rs` carries it)

Invariant #4: *generated, not authored.* Every emitted file opens with a
machine-checkable header so any file's lineage is one `grep` away:

```rust
//! @generated by tesseract-rs-ast-dll-codegen-v1
//! upstream:   tesseract-ocr/tesseract@<full-40-char-sha for tag 5.5.0>
//! ir-snapshot: <sha256 of the ruff_cpp_spo ndjson that produced this file>
//! codegen-plan: tesseract-rs-ast-dll-codegen-v1 @ <plan-commit-sha>
//! DO NOT EDIT — regenerate via the codegen harness; hand-edits are overwritten.
```

- `@generated` is the conventional marker (rustfmt/clippy/review tools key off
  it; CI can skip churn-review on `@generated` files).
- The **three SHAs** (corpus commit, IR snapshot, plan commit) make a file
  reproducible: re-running codegen with the same three MUST be byte-identical
  (this is `CPP-AST-RT` extended to the emit side — D-OCR-41's re-run-diff).

## §2 — Generated-module layout

```
src/
├── lib.rs              # hand-authored shim: `pub mod generated;` + re-exports
├── page_seg_mode.rs    # existing published crate (hand-authored) — untouched
├── generated/
│   ├── mod.rs          # @generated — the module tree (regenerated wholesale)
│   └── <class>.rs      # @generated — one file per transcoded C++ TU/class
```

- `src/generated/` is **wholly owned by the codegen harness** — never
  hand-edited. `mod.rs` is regenerated, so adding/removing a transcoded class is
  a codegen re-run, not a manual `mod` line.
- `lib.rs` stays hand-authored and is the ONLY seam between generated code and
  the published crate surface (it decides what to `pub use`).

## §3 — FFI-wrapper coexistence (no premature deletion)

The existing `src/lib.rs` + `src/page_seg_mode.rs` are the **published
`tesseract-rs` crate** (the `tesseract-sys`/`tesseract-plumbing` FFI wrapper,
v0.15.2). Per the headstone, retiring it is deferred until generated code covers
the surface. During transition:

- Generated code lands under a **`transcode` feature** (off by default), so the
  published FFI crate keeps building and shipping for current consumers.
- The FFI wrapper doubles as the **diff-gate oracle** (D-OCR-42): the generated
  function's output is compared against the FFI call on the same input. So the
  wrapper EARNS its keep during transition — it is the ground truth the
  transcode is validated against, not dead weight.
- Retirement is a **separate operator decision** once `transcode` reaches parity
  on a route; it is never a silent side effect of a codegen run.

## §4 — The receive-contract `tesseract-rs-ast-dll-codegen-v1` must satisfy

When the codegen plan emits into this repo, it MUST:

1. Write only under `src/generated/` (+ regenerate `generated/mod.rs`); never
   touch hand-authored files except via an explicit `lib.rs` re-export PR.
2. Emit the §1 provenance header on every file, with all three SHAs populated
   (no `<...>` placeholders — a missing SHA fails the §5 CI gate).
3. Be **deterministic** — re-running with the same (corpus SHA, IR snapshot,
   plan commit) produces byte-identical output (committed, not build-time
   generated, so reviewers see real Rust — D-OCR-41).
4. Keep generated code behind the `transcode` feature until parity is gated.
5. Vendor **no C++** (Iron Rule #1) — the corpus stays at the §0 pin upstream.

## §5 — CI gate (lands with the first generated file, not before)

A `provenance-header` check (a small script or a `cargo xtask`): every file
under `src/generated/` must (a) start with `//! @generated`, (b) carry three
non-placeholder SHAs, (c) not appear in a hand-edit diff without a matching
codegen re-run. Wire it into the existing `.github/workflows` when §2's
`generated/` first exists — not now (an empty gate is noise).
