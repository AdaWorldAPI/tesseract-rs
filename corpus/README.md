# tesseract-rs golden corpus

Golden corpus for Phase 6 ("E2E validation + performance") of
`.claude/plans/pdf-to-text-ocr-v1.md`, specifically D6.1 (this corpus),
D6.2 (the parity harness that consumes it), D6.3 (perf), and D6.4 (CI
wiring). Nothing here is copied from an external text/document corpus —
every fixture is either generated deterministically from text authored in
this repo (`pages/`, `pdfs/`) or extracted from the Apache-2.0
`eng.traineddata` (`model/`, see `model/README.md`), or is one of the
arc's already-proven synthetic line/page fixtures (`lines/`).

## Layout

```
corpus/
  README.md
  model/    eng.lstm, eng.lstm-unicharset, eng.lstm-recoder,
            eng.lstm-word-dawg, eng.lstm-punc-dawg, eng.lstm-number-dawg, README.md
  lines/    img_8.pgm img_16.pgm img_24.pgm img_40.pgm img_64.pgm img_100.pgm
            line36.pgm page_roomy.pgm page_tight.pgm
  pages/    page_01.pgm .. page_10.pgm  +  page_01.gt.txt .. page_10.gt.txt
  pdfs/     doc_01.pdf .. doc_05.pdf    +  doc_01.gt.txt .. doc_05.gt.txt
  golden/
    lines/  <stem>.txt (our text output per line fixture), line36.dict.txt, line36.dict.tsv
    pages/  page_NN.txt          (our recognize_page_makerow output, dict on)
    pdfs/   doc_NN.txt           (our extract_text_layer output)
    cli/lines/ <stem>.psm13.txt  (banked `tesseract --psm 13` stdout)
    cli/pages/ page_NN.psm6.txt + page_NN.psm6.tsv  (banked CLI outputs)
  gen/      gen_pages.py gen_pdfs.py run_cli_golden.py
```

Notes on what's an *input* vs an *output* in this tree:

- `model/`, `lines/`, `pages/`, `pdfs/` are **inputs** — committed fixture
  bytes the pipeline is run against. `pages/*.pgm` and `pdfs/*.pdf` are
  produced by the two scripts in `gen/`; `lines/*.pgm` are the arc's
  already-proven synthetic line/page fixtures (see Provenance below);
  `model/*` is extracted from the upstream trained model.
- `golden/` is **output** — what our own pipeline (and, for Tier B, the
  `tesseract` CLI) produced when run against those inputs, committed so CI
  can diff against it without re-running anything expensive. It is
  regenerated by the refresh workflow below, never hand-edited.

`pages/*.pgm` are clean digital renders (10 pages, one with added synthetic
noise) of hand-authored text — they are a scanned-page **stand-in**, not
photographs or scans of a real document; nothing here claims otherwise.

## The tier model

Every comparison in the parity harness (D6.2) falls into exactly one of
three tiers. This framing is the plan's D6.2 gate language verbatim — it is
never blurred, because the three tiers have different failure semantics:

- **Tier A — byte gates vs our own goldens (hard CI gates):** `lines/`
  (each fixture's text output was byte-parity-proven against libtesseract
  API-composition oracles when its stage landed: A6b/pixScale/C1/B3-full
  arcs) · `pdfs/` (deterministic lopdf text-layer extraction) · `pages/`
  (determinism + regression of the makerow composition). A golden change is
  a BEHAVIOR change and must be justified in the PR.
- **Tier B — CLI comparison, report only (CER/WER, `golden/cli/` banked):**
  the `tesseract` CLI wraps stages that are NOT part of the transcoded
  chain (DPI estimation on headerless PGMs, invert-retry, ROW-based
  normalization inside PSM 13, ColumnFinder/tospace word segmentation in
  PSM 6), so CLI output can differ while every transcoded leaf is
  byte-parity-proven — measured example: `line36.pgm` ours-with-dict `Ly,`
  vs CLI-psm13 `i,`. The report NAMES the divergent framing; it never
  claims byte parity where the comparison crosses untranscoded stages.
- **Tier C — perf (report, no gate).**

In short: Tier A failing means we broke something we own. Tier B moving
means the untranscoded CLI stages did something different — worth reading,
never a build failure. Tier C is purely informational (D6.3).

## Refresh workflow

1. **Regenerate the input fixtures** (idempotent — rerunning either script
   produces byte-identical output; a diff means an intentional content
   change, e.g. an edited sentence or a layout tweak, not generator drift):

   ```sh
   python3 corpus/gen/gen_pages.py
   python3 corpus/gen/gen_pdfs.py
   ```

2. **Regenerate the Tier A goldens** (orchestrator-only, after a fixture or
   a pipeline-behavior change):

   ```sh
   UPDATE_GOLDEN=1 cargo test -p tesseract-ocr --test golden_lines
   UPDATE_GOLDEN=1 cargo test -p tesseract-ocr --test golden_pages
   UPDATE_GOLDEN=1 cargo test -p tesseract-ocr-pdf --test golden_pdfs
   ```

3. **Refresh the Tier B CLI bank** (`golden/cli/`):

   ```sh
   python3 corpus/gen/run_cli_golden.py
   ```

4. Commit the corpus + golden changes together with the PR that justifies
   the behavior change — never let a golden drift in silently alongside an
   unrelated diff.

## Provenance + license

- `pages/*.pgm`, `pages/*.gt.txt` — text authored directly in
  `gen/gen_pages.py`; no external corpus, no scraped text.
- `pdfs/*.pdf`, `pdfs/*.gt.txt` — text authored directly in
  `gen/gen_pdfs.py`; the PDF bytes are hand-assembled PDF 1.4 (no
  third-party PDF library dependency at generation time).
- `model/*` — extracted from the Apache-2.0-licensed `eng.traineddata`; see
  `model/README.md` for the full provenance, license, and per-file SHA256
  table.
- `lines/*` — the arc's proven synthetic line/page fixtures (A6b, pixScale,
  C1, B3-full arcs — see the top-level `CLAUDE.md`); deterministic bytes,
  generators banked under `.claude/harvest/oracles/` (e.g.
  `gen_page_fixture.py`, `gen_textline_fixtures.py`) — not regenerated by
  the scripts in this `gen/` directory.
